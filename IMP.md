## 1. 总体策略

### 1.1 推进原则
1. **小步快跑**：每 2 周一个可演示版本，避免长周期黑洞
2. **垂直切片**：先跑通"单产线单缺陷"端到端，再横向扩展
3. **数据先行**：每个阶段开始前先备齐数据，避免空跑
4. **人在回路**：所有阶段都包含真实用户验证，不闭门造车
5. **可回滚**：每阶段产出可独立运行，失败不影响前一阶段

### 1.2 阶段总览

| 阶段 | 周期 | 目标 | 关键产出 |
|------|------|------|---------|
| M0 环境搭建 | 2 周 | 跑通基础链路 | Hello World Agent |
| M1 MVP | 10 周 | 单产线单缺陷闭环 | 可演示 Demo |
| M2 多 Agent | 8 周 | 6 角色协作 | 协作可视化 |
| M3 记忆系统 | 6 周 | 三层记忆 + 知识沉淀 | 经验复用 |
| M4 多产线 | 14 周 | 3 产线接入 | 多产线运行 |
| M5 自进化 | 12 周 | 反馈闭环 + 持续优化 | 缺陷率下降 |

跑通 LangGraph + LLM + 工具调用的最小闭环，验证技术栈可行性。

|----|------|------|------|------|------|

| M0-3 | LangGraph Hello World | 写最简 ReAct Agent，能调用 1 个工具 | `agent/hello.py` | 1d | M0-2 |
| M0-4 | MCP 工具原型 | 写 1 个 echo 工具的 MCP Server | `mcp_servers/echo.py` | 1d | M0-1 |
| M0-5 | LangSmith 接入 | 配置 trace，能看到每次 LLM 调用 | trace 可见 | 0.5d | M0-2 |

| M0-7 | Docker 基础 | 写 Dockerfile + docker-compose | 可一键启动 | 1d | M0-1 |

### 2.3 验收标准
- [ ] `docker compose up` 一键启动
- [ ] 输入"查询批次 X 的硬度"，Agent 能调工具并返回结果（哪怕是假数据）
- [ ] LangSmith 能看到完整 trace
- [ ] 代码已上 Git，CI 跑通

### 2.4 风险
- **LLM API 选型纠结**：决策时间 < 1 天，先用通义千问，后续可切换
- **MCP 学习成本**：预留 1 天啃文档

| 产线接口不稳 | 中 | 中 | 降级 + 重试 + 数据缓存 | 接口失败率 > 5% |

| 多产线差异大 | 高 | 中 | 抽象层设计；先做通用再定制 | 产线 C 接入 > 2 周 |
